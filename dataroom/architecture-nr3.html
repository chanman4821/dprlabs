NR3 — Architecture & product

NR3 is a standalone software platform that uses Jarvis to plan, assign, build, check, and repair real software work. It keeps a record of what it did instead of asking you to trust a black box.

Pipeline

Plan → assign → build → check → fix. If it is not good enough, send it back.

The work moves through five jobs. The planner scopes the goal. The dispatcher assigns the task. The builder does the work. The grader checks it. The healer repairs failures. If the grader rejects the output, it sends the task back instead of calling it done.

Planner

Breaks a goal into ordered tasks with clear finish lines before work starts.

Dispatcher

Routes each task to the right worker and refuses half-scoped work.

Builder

Writes the code, docs, and configuration that make the requested system real.

Grader

Checks output against the criteria and sends weak work back for another pass.

Healer

Catches failures, learns from them, and repairs the run where automation can safely recover.

Rejected work goes back

grader → dispatcher means weak work is assigned again, not shipped.

Under the hood

One brain, many specialists, one memory.

NR3 has one planning brain that decides and plans using a top AI model. It hands work to a team of specialist AI workers, drives the coding tools you already use (Copilot, Claude, Codex), and records every assignment, check, and result in one shared memory.

Quality gates

The four enemies every gate exists to kill.

SLOP

Code that compiles but does not do what was asked: placeholder work, fake tests, or plausible nonsense.

HALLUCINATION

Made-up imports, fictional functions, paths that do not exist, or APIs that return what the model wishes.

STALL

Workers that loop, repeat, refuse, or burn tokens without getting closer to checked output.

DRIFT

A small request expanding into a large, off-scope change that no longer matches the original finish line.

Safety by brakes

Autonomy is bounded by hard stops.

NR3 is designed around brakes, not blind trust: a kill-switch, budget cap, concurrency cap, canary checks, and hard process containment. Those controls let NR3 run many workers without pretending every worker is safe by default.

Why it is hard to copy

The learning loop runs on top of whichever AI models are best.

decide → act → real outcome → calibrate → sharper next decision

This is why it is hard to copy. NR3 records what actually happened, ties the result back to the decision, and uses that memory to make the next assignment, plan, and grade sharper. Because NR3 runs on top of Copilot, Claude, Codex, or whichever AI models are best, it can improve when the models improve.

Honest status

The front half runs today: decide → act → grade is active on real work. The full closed learning loop and hands-off self-repair are the active build focus. A senior human still approves the calls that matter. NR3 claims no autonomy it does not yet have.
